Regions

* We're hoping to make this easier with regions and exclusions. * Regions is a feature that allows you to define areas to flow content through.
* In this diagram, you can see two pieces of content flowing through two different sets of regions. * The first set of content flows through regions 1, 2, 3, and 4. * The second, regions a and b. 
.content { flow-into: story-flow; } 
.region  { flow-from: story-flow; }
* The markup would look something like this. * We take some content, and put it into a flow. * And then we specify that our regions should lay out content from that flow.

Exclusions

* While regions specify containers to flow content through, * Exclusions specify how you can customize the shapes of those containers * As content wraps inside and around them.
* For example, we can wrap content inside and around a circular shape. * Similar to what we saw on the tvsafety site. 
.circle { 
    wrap-flow: end; 
     shape-outside: circle(50%, 50%, 50%); 
     shape-inside: outside-shape; 
 }
* We specify that we want content to wrap on the end (or right) side * Respecting a circle that is centered at 50%, 50%, with a 50% radius * And content inside the block should respect the same shape as content outside.

Standards Bodies

W3C publishes documents that define Web technologies. These documents follow a process designed to promote consensus, fairness, public accountability, and quality. At the end of this process, W3C publishes Recommendations, which are considered Web standards.

W3C FAQ

* Then the question becomes how do we get there. * The W3C has a more focused, technical definition. * They issue recommendations that have undergone a formal review process, and * Those recommendations are web standards.

The W3C Process

Working Draft

Candidate Recommendation

Recommendation

* The W3C recommendation process takes a specification through three major steps * A 'working draft' stage where the specification is available for public comment and can undergo significant revisions. * A 'candidate recommendation' stage where the specification may only undergo minor revisions, and must be implemented by at least two browsers to prove itself. * A 'recommendation' stage, where the specification has been finalized.

A Brief History of CSS

* In order to understand how the W3C Recommendation process works, it's helpful to take a look at the history of CSS. * The specification process has historically taken longer to finalize with each version of CSS. * Here's a brief history.
  • 1990 First web page & web browser
  • 1994 The W3C is formed to help shape standards on the web
  • 1995 CSS1 First public working draft
  • 1996 CSS1 Recommendation
  • 1997 CSS2 First public working draft
  • 1998 CSS2 Recommendation
  • 2000 CSS3 Roadmap
  • 2002 CSS2 Revision 1 Working draft
  • 2011 CSS2 Revision 1 Recommendation
  • 2011 Regions and Exclusions first public working drafts
* Read timeline

CSS 2.1

  • CSS2 lacked a Candidate Recommendation phase
  • Specification and implementations differed
  • Some material was removed
  • Some material was added
* The original plan was to move from CSS2 to CSS3 * But, as the errata list grew for CSS2, the W3C decided to just create a revision * There was no CR phase when CSS2 became a Recommendation, and therefore * The implementability was not as thoroughly tested * Browsers differed from the specification, and 'standards' needed to be what people could use on the web * CSS2.1 aligned the specification and implementation, but it was a long process

Standards synthesize specification & implementation

* Standards are really about evolving browsers and specifications together * If browsers get off track, you may wind up generating tons of browser-specific hacks to get an effect. * If specifications get off track, you may wind up with a document that no one implements.

The W3C helps guide this process

* The W3C is a body that helps guide this * With the CR stage, it allows an opportunity for browsers to grow alongside specifications. * It has the odd role of both documenting behaviors today, and brainstorming ideas for tomorrow.

Evolving Browsers

* As I mentioned before, browsers aren't waiting for specs to finalize to start implementing functionality. * They are continuously working on new features, even some that may still be experimental.

Polyfills

fake the functionality of the future, today

* One workaround we are seeing these days is the use of polyfills. * Polyfills allow you to access future functionality, even when the browser hasn't implemented it yet.

polyfill.js

older browsers

* Typically, polyfills work by filling in missing functionality in a browser with javascript. * You include a javascript library, use the unimplemented feature, and the javascript will transform it into something the older browser can understand. * Traditionally, these were a way of retrofitting legacy browsers with modern standards.

polyfill.js

new functionality

* What we are seeing today, however, is polyfills coming out for features before they have been fully implemented in any browser. * The spec has generally settled to some degree, meaning there may be some partial implementations.

Polyfills

  • work now
  • across browsers
  • can be replaced by native
  • let you try out new features
CodePen Regions Rodeo
* Polyfills allow you to get functionality into people's hands today. * Which means you can get people trying out your feature in something like this. * CodePen, an html sample sharing site, had a series of week-long challenges to see what kind of awesome work people could come up with. * Because Regions had a polyfill, we were able to have one of these challenges be test driving regions and coming up with something awesome. * Early feedback is invaluable.

But

  • Polyfills are a workaround
  • Feature specs are likely to change
  • Polyfills can be non-performant
* It is important to note that there are some tradeoffs to using polyfills. * Most importantly, they are a workaround and require external js libraries. * They also tend to start emerging when specifications are likely to change. * So you may write code that needs to be altered to use native functionality when it comes along. * And finally, because they are running client-side to make adjustments, they can suffer from some performance concerns.
